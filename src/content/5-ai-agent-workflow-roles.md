> **【一言蔽之】**  
> 想象写一份策划案，如果你只让一个 AI 给你建议，它多半会当个“老好人”（全是伟光正的废话）。但在 Tri-Agent 协议里，你放进去：一个**造房子的泥瓦匠（Builder）**，一个**专门挑刺砸墙的黑粉（RedTeam）**，还有一个**整天想着如何坐火箭搬家的玄学家（Explorer）**。他们三个互相抬杠，最后给出的方案绝对是打磨得发亮的真金。

---

## 🔬 一、 核心原理解析 (理解层)

三智能体异步对抗协议（Tri-Agent Protocol）打破了传统 LLM 的顺从属性。它的核心逻辑在于**强制赋予不同的 System Prompt 立场，并剥夺它们之间的共情**。

### 1. 🟢 Builder（建设者/泥瓦匠）
- **核心职能**：正向增强。
- **运行逻辑**：拿到一份概念稿后，寻找闪光点，并提供具体的、主流的、可以直接落地（How-to）的建议。**严禁挑刺。**

### 2. 🔴 RedTeam（破坏者/黑粉）
- **核心职能**：寻找逻辑漏洞与致命缺陷进行压力测试。
- **运行逻辑**：它拿不到原始文本，它只能拿到 Builder 刚写完的方案。它的任务就是疯狂攻击 Builder 方案中隐患大、成本高的地方。

### 3. 🔵 Explorer（探索者/玄学家）
- **核心职能**：挑战第一性原理，提出“如果这一切都不成立呢”的颠覆性问题。
- **运行逻辑**：负责打破所有现成框架的束缚。如果是改进一辆马车，Builder 想要更快的马，RedTeam 考虑马可能生病，而 Explorer 会质问：为什么非要是马？为什么不能用蒸汽机？

## 🛠️ 二、 工程实践指南 (实践层)

这是我们如何在开发中**组装**这三套指令的标准流转过程：

```python
# 伪代码：三级跳协作框架
def process_proposal(draft_doc):
    # 步骤一：建设者给出优化提案，标记为P0/P1
    builder_output = llm(prompt=PROMPT_BUILDER, input=draft_doc)
    
    # 步骤二：破坏者疯狂注入致命攻击向量
    redteam_output = llm(prompt=PROMPT_REDTEAM, input=builder_output)
    
    # 步骤三：探索者跳出棋盘，寻找降维打击
    explorer_output = llm(prompt=PROMPT_EXPLORER, input=draft_doc + redteam_output)
    
    return [builder_output, redteam_output, explorer_output]
```
（在实战中，最后我们会用一个「仲裁者 Agent」将这三份吵得不可开交的决议进行汇总融合。）

## 💡 三、 避坑与进阶 (升华层)

- **最大的坑：立场崩塌 (Role Bleeding)**：如果你使用较弱的千亿参数以下模型进行对抗，很多时候写实着写实着，RedTeam 就变得极其客气（“虽然有问题但是可以理解云云”）。**必须在系统提示词里加上极其强烈的极性词**，甚至威胁它“如果你同意对方，程序将会退出”，以保证针锋相对的张力。
- **对抗边界**：必须约束 RedTeam 不能“凭空捏造”尚未发生的风险，所有的攻击必须基于原文本现有的数据与逻辑。
